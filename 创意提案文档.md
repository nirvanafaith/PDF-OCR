# PDF-OCR：可视化字符级 PDF OCR 精修工具

> 一款为扫描书页 PDF 设计的字符级 OCR 桌面校对工具，提供「画框选区 → 字符级 OCR → 纵校（按字符分组）→ 横校（按行校对）→ 精修输出」的完整可视化工作流。
>
> TRAE AI 创造力大赛参赛作品

## 一、项目简介

PDF-OCR 是一款面向扫描书页 PDF 的字符级 OCR 桌面校对工具。它不同于传统只输出整段文本的 OCR 软件——传统 OCR 给你一段「看上去差不多」的文字，你要么全盘接受，要么在文本编辑器里手动逐字核对，效率极低且容易出错。PDF-OCR 把 OCR 识别结果与原始扫描图像紧密结合，让用户能够在视觉上下文中逐字符地校对、修改、删除、补全，最终输出一份经过人工精修的高质量 PDF 文本层。

项目采用「四阶段流水线」设计：**导入 → 纵校 → 横校 → 精修**。每个阶段都对应一个独立的窗口，由主窗口（MainWindow）通过 QStackedWidget 按阶段切换，阶段之间通过 Qt 信号传递数据。这种设计让用户可以聚焦当前阶段的工作，不被其他阶段的复杂性干扰；同时支持阶段间回退（带 back_signal），允许用户在发现上游问题时随时返回修正。

整个项目用 **TRAE IDE** 开发，全程借助 TRAE 进行代码理解、bug 定位、重构建议、性能瓶颈分析和技术报告撰写。项目由 software1（PyQt6 + RapidOCR + PaddleOCR PP-OCRv5）和 software2（PyQt6 + PyMuPDF + reportlab）两个 Python 版本组成，software1 负责画框 + OCR 推理，software2 负责数据校对与精修输出，两者通过 chars.json / lines.json 串联，是一次「研究 → 工程」的完整闭环实践。

## 二、创意来源与痛点

### 2.1 传统 OCR 工具的痛点

扫描书页 PDF 的字符级 OCR 是一个长期被忽视的难题。市面上主流 OCR 工具（无论是商业的还是开源的）普遍存在以下痛点：

1. **只给文本，不可精修**：OCR 引擎输出一段识别文本，用户复制出来后要在文本编辑器里逐字核对，看不到原文上下文，错字很难发现。
2. **行级识别，丢失字符级信息**：大多数 OCR 引擎以「行」为最小单位输出，单字置信度、字符边界框等信息被丢弃。哪怕引擎本身识别错了某个字，用户也无法定位到具体哪个字错了。
3. **跨页同字错误难修正**：扫描书页里同一个常用字（如「的」「了」「是」）在全文可能出现上千次，OCR 引擎如果对某个字识别有系统性偏差（例如把「己」识别成「已」），用户需要在每一页单独修正，重复劳动极大。
4. **大 PDF 翻页卡顿**：扫描书页 PDF 动辄数百页，全量渲染会撑爆内存，翻页时延迟严重。
5. **缺乏视觉反馈**：识别结果与原图脱离，用户无法直观地看到「OCR 在哪个区域识别错了」。

### 2.2 PDF-OCR 如何解决这些痛点

PDF-OCR 针对每一个痛点设计了对应的功能：

| 传统痛点 | PDF-OCR 的解决方案 |
|---------|-------------------|
| 只给文本不可精修 | 四阶段流水线 + 精修窗口，支持拖拽、8 手柄缩放、右键删除等精细操作 |
| 行级识别丢失字符信息 | 字符级 OCR，每个字符都有独立的边界框、置信度、ID |
| 跨页同字错误难修正 | 纵校阶段按字符分组，把全文中同一个字聚合在一起批量校对 |
| 大 PDF 翻页卡顿 | LazyPageLoader 懒加载 + LRU 缓存，按 (page, zoom) 缓存渲染后页面 |
| 缺乏视觉反馈 | 横校阶段将识别文本叠加在原图上显示，错误一目了然 |

## 三、核心功能与亮点

### 3.1 可视化画框选区

用户在 PDF 页面上用鼠标画框选区，OCR 只处理选区内的内容，避免页眉、页脚、水印、批注等无关内容干扰识别结果。这一功能在 software1（Python）中由 `DrawBoxWindow` 实现，让用户对 OCR 的范围有完全的掌控。

**价值**：传统 OCR 引擎对整页做识别，水印、装订线、批注都会污染结果。画框选区让 OCR 聚焦在真正需要识别的正文区域，识别准确率与处理速度都显著提升。

### 3.2 字符级 OCR

不仅识别文本行，还能识别单个字符并保存字符级元数据：

```json
{
  "char_id": 0,
  "line_id": 0,
  "page_num": 0,
  "char": "智",
  "score": 0.99,
  "box": [x1, y1, x2, y2]
}
```

每个字符都有独立的 `char_id`、`line_id`、`page_num`、`char`、`score`、`box` 字段。这意味着用户后续可以定位到「第 3 页第 7 行第 4 个字」这种粒度进行精修。

**技术实现**：software1 使用 RapidOCR + PaddleOCR（PP-OCRv5），通过 `return_single_char_box=True` 参数获取单字符边界框。OCR 引擎支持 CUDA GPU 加速，对大文档批量识别时推理速度可提升数倍。

### 3.3 纵校模式（按字符分组校对）

这是 PDF-OCR 最具创意的功能之一。纵校阶段把全文中**同一个字符**的所有出现聚合到一起，让用户一次性校对同字的所有实例。

**举例**：如果 OCR 引擎对「己」字有系统性偏差，常常识别成「已」，用户在纵校模式下选择「己」字符组，会看到全文中所有「己」的实例列表（每个实例都附带原图切片预览）。用户只需修改一次，所有实例同步更新，避免逐页重复劳动。

**交互细节**：
- 左侧字符列表：按字符聚合，显示该字符在全文出现的次数
- 中间预览区：选中字符后，显示该字符的所有原图切片（带边界框）
- 右侧修改区：支持修改文字、新增多字符、删除实例
- Ctrl + 滚轮缩放、按住左键拖拽平移预览图

**数据流**：纵校窗口直接修改 `char_slices`（字符切片字典）与 `ocr_results`（OCR 结果），完成后通过 `finished_signal` 将更新后的数据传递给横校阶段。

### 3.4 横校模式（按行校对）

横校阶段按页逐行展示识别文本与原图叠加视图。用户在每页原图上看到识别结果叠加在对应位置，错误一目了然。

**交互细节**：
- 工具栏：上一页 / 下一页 / 页码跳转 / 缩放控制
- 主视图：原图 + 识别文字叠加（QGraphicsPixmapItem + QGraphicsTextItem）
- 鼠标悬停高亮当前行
- 双击行打开「修改文字」对话框
- 右键菜单：忽略该行、修改文字
- Ctrl + 滚轮缩放整个视图

**性能优化**：横校阶段使用 pixmap 缓存避免重复缩放，缓存键为 `(page, zoom_level)`。悬停预览使用单条目缓存（`slice_cache_id_` + `slice_cache_pixmap_`），避免重复生成切片图像。

### 3.5 精修窗口（拖拽 / 缩放 / 删除）

精修阶段是流水线的最后一站，用户在此对横校输出的行文本做最终精细调整，并导出为 PDF。精修窗口支持以下操作：

- **拖拽移动**：鼠标按住文本块拖拽到正确位置
- **8 手柄缩放**：选中文本块后，四角 + 四边共 8 个手柄，拖拽手柄调整文本块大小
- **右键删除**：右键文本块弹出菜单，删除不需要的文本块
- **双 PDF 输出**：导出两份 PDF——红色文字版（用于人工核对）+ 透明文字版（用于机器后续处理，可作为可搜索 PDF 的文本层）

**输出技术**：使用 reportlab 生成红色/透明两种嵌字 PDF。

### 3.6 双版本实现（software1 + software2）

项目由两个 Python 版本组成，覆盖不同场景：

| 版本 | 目录 | 技术栈 | 角色 |
|------|------|--------|------|
| software1 | `software1/` | PyQt6 + RapidOCR + PaddleOCR + PyMuPDF + Surya | 基础版，含完整 OCR 推理（含画框选区） |
| software2 | `software2/` | PyQt6 + PyMuPDF + reportlab | 增强版，专注数据处理（导入 → 纵校 → 横校 → 精修），不做 OCR 推理 |

**为什么需要双版本？**
- software1 验证 OCR 推理的可行性（含 GPU 加速）
- software2 把 OCR 推理与数据校对解耦，OCR 结果以 JSON 形式导入，让校对工具可以独立运行（适合 OCR 已由 MinerU 等其他工具预处理过的场景）

### 3.7 GPU 加速

software1 的 OCR 引擎支持 CUDA 加速推理：

- 通过 RapidOCR 的 ONNX Runtime 后端切换 GPU/CPU 执行
- 自动处理 cuDNN 路径（详见 `.trae/specs/fix-gpu-cudnn-path.md`）
- 支持批量 OCR 识别（详见 `.trae/specs/improve-gpu-utilization-parallel-ocr.md`），提升 GPU 利用率

**价值**：扫描书页 PDF 动辄数百页，纯 CPU OCR 推理可能耗时数十分钟，GPU 加速可缩短至数分钟，让交互式校对成为可能。

### 3.8 懒加载与流畅翻页

针对大 PDF 的卡顿问题，项目实现了 `LazyPageLoader`：

- **按需渲染**：PDF 页面在首次访问时才渲染为图像，避免一次性渲染所有页撑爆内存
- **LRU 缓存**：维护固定大小的缓存（默认 5 页），最近访问的页面保留在内存，淘汰最久未访问的页
- **懒加载优化**：LazyPageLoader 按 (page, zoom) 缓存，避免重复渲染；纵校切换字符组时增量更新而非全量重建

**价值**：让数百页的扫描书 PDF 在普通办公电脑上也能流畅翻页校对，无需高端工作站。

## 四、操作方式与使用流程

PDF-OCR 的使用流程分为四个阶段，每个阶段对应一个独立的窗口：

### 阶段 1：导入（ImportWindow）

1. 启动程序，主窗口顶部显示步骤指示器：`导入 → 纵校 → 横校 → 精修`，当前阶段高亮
2. 在导入窗口选择 PDF 文件（software2 还支持直接导入已识别的 JSON 结果）
3. 后台线程（`ImportWorker` 使用 worker object pattern）异步渲染 PDF 为页面图像，主线程不卡顿
4. 渲染完成后，通过 `finished_signal` 把 `page_images`、`ocr_results`、`char_slices` 传递给主窗口

### 阶段 2：纵校（VerticalCheckWindow）

1. 主窗口切换到纵校窗口，步骤指示器更新为当前阶段
2. 左侧字符列表显示全文中所有出现过的字符，按出现频率排序
3. 点击某个字符（如「的」），中间预览区显示该字符在全文所有出现位置的原图切片
4. 检查切片，发现 OCR 错误时在右侧修改区修改文字（可修改为多字符，如把「的」改成「目的」）
5. 修改后所有相关实例同步更新
6. 完成纵校后点击「下一步」，进入横校阶段

### 阶段 3：横校（HorizontalCheckWindow）

1. 主窗口切换到横校窗口
2. 主视图按页显示：原图 + 识别文字叠加
3. 鼠标悬停高亮当前行，双击打开「修改文字」对话框
4. 右键菜单选择「忽略该行」或「修改文字」
5. 工具栏切换页面、调整缩放
6. 完成横校后点击「下一步」，进入精修阶段

### 阶段 4：精修（RefineWindow）

1. 主窗口切换到精修窗口
2. 显示横校输出的行文本块，鼠标拖拽调整位置
3. 选中文本块，8 个手柄出现，拖拽手柄调整大小
4. 右键文本块删除不需要的实例
5. 点击「导出 PDF」按钮，生成两份 PDF：
   - **红色文字版**：识别文字以红色叠加在原图上，便于人工核对
   - **透明文字版**：识别文字作为透明文本层嵌入 PDF，可作为可搜索 PDF 使用
6. 导出完成后弹出成功对话框，显示两个 PDF 文件的路径
7. 完成！可重新开始新一轮处理

### 阶段间回退

任何阶段都可以通过「返回」按钮回到上一阶段（携带原数据重建上游窗口），方便用户在发现上游问题时随时修正。

## 五、技术架构

### 5.1 整体架构

两个版本均采用 **主窗口 + 阶段窗口（QStackedWidget）** 的架构：

- **MainWindow** 持有 `QStackedWidget`，按阶段切换子窗口
- **StepIndicator** 顶部显示当前阶段，已完成阶段显示为绿色，当前阶段显示为蓝色
- 阶段窗口之间通过 Qt 信号传递数据，窗口间无直接引用（低耦合）

### 5.2 数据流

```
ImportWindow
   │ (page_images, ocr_results, char_slices)
   ▼
VerticalCheckWindow  ── 按字符分组校对
   │ (updated char_slices, updated ocr_results)
   ▼
HorizontalCheckWindow  ── 按行校对
   │ (corrected_lines)
   ▼
RefineWindow  ── 精修与输出
   │ (red_pdf_path, transparent_pdf_path)
   ▼
   完成
```

### 5.3 模块划分

| 模块 | software2 | 职责 |
|------|-----------|------|
| PDF 处理 | `pdf_processor/pdf_loader.py` | PDF → 图像 |
| OCR 引擎 | `ocr_engine/rapidocr_engine.py` | JSON 加载、字符分组、行数据构建 |
| PDF 输出 | `pdf_processor/pdf_output.py` | 校对后 PDF 生成 |
| 数据模型 | `models/data_models.py` | dataclass 定义 |
| UI 主窗口 | `main.py` | 阶段调度 |
| UI 阶段窗口 | `ui/*.py` | 各阶段交互界面 |
| 样式 | `ui/styles.py` | QSS 样式表 |
| 缩放工具 | `ui/zoom_utils.py` | 滚轮缩放计算 |
| 懒加载 | `pdf_processor/pdf_loader.py`（software1/software2） | 大 PDF 按需渲染 |

### 5.4 线程模型

| 场景 | software2 |
|------|-----------|
| PDF 导入 | `ImportWorker(QObject)` + `moveToThread()` ✓ |
| PDF 输出 | `PDFOutputWorker(QThread)` 继承 `run()` ✗（违反最佳实践） |
| 横校数据构建 | 主线程同步（大文档卡顿） |

### 5.5 数据持久化

项目使用 JSON 文件作为持久化存储，核心数据文件为 `lines.json`（行记录）与 `chars.json`（字符记录）。经评估，两者均满足 BC 范式（BCNF）——无传递依赖、部分依赖，所有非主属性完全依赖于候选键。

- **lines.json 候选键**：`(page_num, line_id)`
- **chars.json 候选键**：`(page_num, line_id, char_id)`

### 6.2 Bug 定位与修复

项目过程中借助 TRAE 定位并修复了多个关键 bug，部分示例：
- **横校 pixmap 缓存被整体替换**：缓存字典被整体赋值导致缓存失效，每次翻页都重新缩放整页图像——TRAE 帮助识别为单字符笔误（`= {}` 应改为 `[key] =`）
- **横校 pixmap 缓存被整体替换**：切换页面时 `self._pixmap_cache = {cache_key: scaled_pixmap}` 整体替换而非增量更新，导致缓存失效——TRAE 帮助定位并改为增量更新 `self._pixmap_cache[cache_key] = scaled_pixmap`
- **OCREngine 死代码引用未初始化属性**：software2 不做 OCR 推理但保留了引用 `self.engine` 的死代码——TRAE 帮助识别为潜在崩溃风险
- **GPU cuDNN 路径问题**：CUDA 加速推理时 cuDNN 找不到——TRAE 协助配置环境变量与库路径

### 6.3 重构建议

TRAE 在代码审查中给出了多项重构建议，例如：
- 将 `PDFOutputWorker` 从「继承 QThread 重写 run()」改为「worker object pattern」（`QObject + moveToThread`），与 `ImportWorker` 风格统一
- 将横校 pixmap 缓存从整体替换改为增量更新，避免缓存失效；UI 层操作 JSON 的代码封装到 `OCREngine`
- 抽取 software1 / software2 重复的 `pdf_loader.py`、`styles.py`、`zoom_utils.py` 为共享模块

### 6.4 性能瓶颈分析

借助 TRAE 完成性能瓶颈识别与优化方案设计：
- 识别 `build_line_data` 主线程同步阻塞问题，设计基于 QThread worker object pattern 的异步化方案
- 识别横校 pixmap 缓存键设计问题（缩放级别变化即缓存失效），设计多级缓存策略
- 识别 OCREngine 中 `_recognize_page` 和 `_recognize_page_batch` 两个死代码方法引用未初始化的 `self.engine`，删除以降低维护成本
- 识别纵校 `flush_pending_modifications` 全量重建字符列表的低效问题，设计增量更新方案

## 七、适用场景

PDF-OCR 适用于任何需要字符级精度的 PDF OCR 场景：

- **古籍数字化**：纵校模式特别适合批量修正古籍中频繁出现的同一字符的 OCR 错误
- **教辅书扫描件转文本**：横校模式让识别结果叠加在原图上，错误一目了然
- **学术论文 OCR**：精修窗口支持精细调整数学公式、特殊符号位置
- **档案数字化**：双 PDF 输出（红色文字版 + 透明文字版）便于人工核对与机器后续处理
- **批量 OCR 校对**：纵校模式让用户一次性校对全文中同字的所有出现，效率倍增
- **大文档处理**：懒加载与 LRU 缓存让数百页 PDF 也能流畅翻页
